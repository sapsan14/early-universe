# ARCHEON — Инструкции для LLM-агентов

Этот документ — единый источник правды для любого LLM-агента, работающего с проектом ARCHEON. Прочитай его полностью перед началом работы. Возвращайся сюда при любой неясности.

---

## 1. Миссия проекта

ARCHEON (**A**rchaeology of the **R**adiation-dominated **C**osmos via **H**ybrid **E**mulation and **O**bservational **N**eural inference) — открытая вычислительная платформа для:

- Физически строгой симуляции ранней Вселенной
- Нейросетевого инференса космологических параметров из CMB-данных (обратная космология)
- Обнаружения аномалий в CMB (Cold Spot, негауссовость)
- Интерактивной визуализации через WebGPU-обсерваторию в браузере

Цель: стать новым стандартом на стыке космологии, машинного обучения и интерактивных научных инструментов. Конечная точка: публикация на arXiv + PyPI-пакет + демо-сайт.

## 3. Правила документирования

### 3.1 Принцип: тщательно, но без избыточности

- Каждый модуль, класс и публичная функция **обязаны** иметь docstring
- Docstrings содержат: назначение, параметры с типами, возвращаемое значение, формулу (если есть)
- **Не** создавай отдельные README для подпапок — вся документация в docstrings + PLAN.md + KNOWLEDGE.md
- **Не** создавай отдельные файлы для changelog, contributing guide, или code of conduct, пока не запрошено явно
- Комментарии в коде — только для неочевидной логики. Не комментируй то, что и так видно из кода

### 3.2 Ссылки на KNOWLEDGE.md

**Правило первого упоминания:** при первом упоминании термина в любом документе проекта (.md файл) ставь ссылку на KNOWLEDGE.md:

```markdown
Решатель [уравнений Фридмана](KNOWLEDGE.md#friedmann-equations) вычисляет эволюцию
[масштабного фактора](KNOWLEDGE.md#scale-factor) a(t) для заданных
[Omega_m](KNOWLEDGE.md#omega-m), [Omega_Lambda](KNOWLEDGE.md#omega-lambda) и [H0](KNOWLEDGE.md#h0).
```

В коде (docstrings): используй формат `See KNOWLEDGE.md#friedmann-equations` вместо полной ссылки.

### 3.3 Добавление новых терминов

Если в процессе работы появляется новый термин, концепция или формула:

1. **Сначала** добавь запись в KNOWLEDGE.md (в соответствующий раздел A-K)
2. Используй стандартную структуру: Интуиция, Формула, Подробнее, В ARCHEON, Связано
3. **Затем** ссылайся на неё из кода/документации

### 3.4 Формулы

- В KNOWLEDGE.md и PLAN.md: LaTeX формат `$$ ... $$` (блочные) и `$ ... $` (инлайн)
- В docstrings Python: ASCII-совместимый формат: `H(a) = H0 * sqrt(Omega_r/a^4 + Omega_m/a^3 + Omega_Lambda)`
- Каждая формула в KNOWLEDGE.md сопровождается расшифровкой всех символов

## 4. Стандарты кода

### 4.1 Python

```python
"""Модуль для решения уравнений Фридмана на JAX.

See KNOWLEDGE.md#friedmann-equations for the theoretical background.
"""

import jax.numpy as jnp
from jax import jit


@jit
def hubble_parameter(
    a: jnp.ndarray,
    H0: float,
    Omega_m: float,
    Omega_r: float,
    Omega_Lambda: float,
) -> jnp.ndarray:
    """Compute the Hubble parameter H(a).

    H(a) = H0 * sqrt(Omega_r/a^4 + Omega_m/a^3 + Omega_Lambda)

    See KNOWLEDGE.md#friedmann-equations

    Args:
        a: Scale factor(s). a=1 today, a<1 in the past.
        H0: Hubble constant in km/s/Mpc.
        Omega_m: Total matter density parameter.
        Omega_r: Radiation density parameter.
        Omega_Lambda: Dark energy density parameter.

    Returns:
        H(a) in the same units as H0.
    """
    return H0 * jnp.sqrt(Omega_r / a**4 + Omega_m / a**3 + Omega_Lambda)
```

**Обязательно:**
- Type hints на всех аргументах и возвращаемых значениях
- Docstring на каждой публичной функции/классе (Google style)
- Ссылка `See KNOWLEDGE.md#anchor` для функций, реализующих физику/ML
- `@jit` для вычислительно тяжёлых функций на JAX
- Осмысленные имена переменных: `scale_factor`, не `sf`; `Omega_m`, не `om`

**Запрещено:**
- Однобуквенные переменные (кроме общепринятых: `a` для scale factor, `k` для wavenumber, `l` для multipole, `z` для redshift)
- `import *`
- Комментарии, дублирующие docstring

### 4.2 TypeScript (Web)

- Строгие типы, никаких `any`
- React functional components
- JSDoc для публичных функций

### 4.3 Тесты

- Каждый физический модуль имеет тест в `archeon/tests/`
- Валидационные тесты: сравнение с CLASS/CAMB с допуском < 0.1%
- ML тесты: проверка размерности, smoke test на одном батче
- Формат: `pytest`

## 5. Конвенции именования

### 5.1 Файлы и директории

- Python модули: `snake_case.py` (`friedmann.py`, `bayesian_cnn.py`)

- Физические константы: `UPPER_SNAKE_CASE` в отдельном файле `archeon/physics/constants.py`
- Значения Planck 2018: конфиг `archeon/config/planck2018.yaml`

## 6. Процесс работы

2. Следуй стандартам кода (раздел 4)
3. Ссылайся на KNOWLEDGE.md при первом упоминании термина в документации
4. Если появляется новый термин — сначала в KNOWLEDGE.md, потом в код

1. Обнови статус задачи в таблице PLAN.md: `pending` -> `done`
2. Если добавлены новые концепции — проверь, что они в KNOWLEDGE.md
3. Запусти тесты (если применимо)

## 7. Приоритеты

При конфликте приоритетов:

1. **Физическая корректность** > скорость > удобство. Результат должен быть научно валидным.
2. **Простота** > сложность. Не усложняй без необходимости. Один файл лучше трёх.
3. **Воспроизводимость** > производительность. Каждый запуск должен быть воспроизводим (seed, конфиги, версии).
4. **Документация = код**. Недокументированный код считается незавершённым.
